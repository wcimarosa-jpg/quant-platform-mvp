# Analytic Compatibility — Questionnaire Design Constraints

## Purpose

Every questionnaire generated by this system must be compatible with the analysis pipeline in `survey_analysis_toolkit/`. This document defines the hard constraints a generated survey must satisfy so that each planned analysis can run without manual data reformatting.

The agent must consult this file **before** generating any questionnaire to ensure the output is machine-analyzable.

---

## Constraint Table (Quick Reference)

| Planned Analysis | Questionnaire Requirement |
|---|---|
| Cross-tabs (single select) | Exhaustive, mutually exclusive response codes with `value_labels` |
| Cross-tabs (multi select) | Binary-coded items sharing a `var_prefix` (e.g., `QB2r1`, `QB2r2`) |
| Cross-tabs (T2B summary) | Rating batteries on a consistent Likert scale; `t2b_codes` must be defined |
| Cross-tabs (mean battery) | Items on a uniform numeric scale (all same range) |
| K-Means Clustering | 15–40 Likert attitude/motivation items on a uniform scale (5pt or 7pt) |
| VarClus | Same battery as K-Means; 10+ items minimum |
| MaxDiff (HB Estimation) | Comparable items within a single evaluative dimension; BIB design parameters |
| MaxDiff Head-to-Head | Paired a/b variant items included in the MaxDiff design |
| MaxDiff Bundle Evaluation | Pre-defined item bundles (3–5 items each) from the MaxDiff item set |
| MaxDiff Regression | External IV columns (demographics, attitudes) alongside MaxDiff tasks |
| Ridge Regression (Multi-DV) | Clear IV battery (attitudes) and DV battery (behaviors) on numeric scales |
| Segment Banner | One variable with clean integer-coded groups usable as `segment_var` |

## Detailed Constraints by Analysis Type

### 1. Cross-Tabulation & Data Tables

**Module:** `tables.py` via `generate_full_data_tables()` / `generate_segment_crosstabs()`

**Single-select questions:**
- Every response option must have a unique integer code
- Codes must be exhaustive (cover all valid responses) and mutually exclusive
- Config requires `value_labels = {1: 'Male', 2: 'Female', 3: 'Non-binary'}`
- Include codes for "Prefer not to say" / "Other" if used — never leave uncoded options

**Multi-select questions:**
- Each item must be a separate binary column: 1 = selected, 0 = not selected
- Columns must share a prefix: `QB2r1`, `QB2r2`, `QB2r3`, etc.
- Config requires `item_labels = {'QB2r1': 'Instagram', 'QB2r2': 'TikTok', ...}`

**Rating batteries (T2B):**
- All items in a battery must use the same scale (e.g., 5-point agree/disagree)
- Scale must be numeric and ordered (1 = Strongly Disagree → 5 = Strongly Agree)
- Config requires `t2b_codes` — the scale points that count as "top-2-box"
  - For a 5-point scale: `t2b_codes = [4, 5]` (Agree + Strongly Agree)
  - For a 7-point scale: `t2b_codes = [6, 7]`
- Items share a `var_prefix`: `QA1r1`, `QA1r2`, etc.

**Mean batteries:**
- Same uniform-scale requirement as T2B batteries
- All items must use identical numeric coding (same min/max)
- Mixed scales within a battery will produce misleading means

**Questionnaire rule:** Every closed-ended question must define its response codes at design time. Open-ended verbatim questions are excluded from cross-tabs.

### 2. K-Means Clustering

**Module:** `clustering.py` via `run_kmeans_sweep()`

**Attitudinal battery requirements:**
- Minimum 15 items, recommended 25–40 items
- All items on the same Likert scale (5-point or 7-point; do not mix)
- Items should be directionally coded: higher = more agreement
- Reverse-coded items must be flagged so the mapper can flip them before analysis
- Items must span multiple conceptual dimensions (not all measuring the same construct)

**Why these constraints matter:**
- K-Means operates on Euclidean distance — mixed scales distort distances
- VarClus runs first to reduce dimensionality; it needs a homogeneous numeric matrix
- Fewer than 15 items lacks variance for meaningful cluster separation
- Unidimensional batteries produce trivial "high vs. low" clusters

**Questionnaire rule:** Include a single large attitudinal battery (one question block) with 20–35 agree/disagree statements covering behavioral attitudes, motivations, values, and category-specific beliefs. Use a consistent 5-point or 7-point scale throughout.

---

### 3. VarClus (Variable Clustering)

**Module:** `clustering.py` via `run_varclus()`

**Input requirements:**
- Same battery used for K-Means clustering
- Minimum 10 items (15+ recommended)
- Numeric columns only — no categorical variables
- Config requires `vars_list` (column names) and `var_labels` (human labels)

**Questionnaire rule:** No additional questionnaire requirements beyond K-Means. VarClus is a preprocessing step that selects representative variables from the attitudinal battery.

---

### 4. MaxDiff (Hierarchical Bayes Estimation)

**Module:** `maxdiff.py` via `run_maxdiff_analysis()`

**Design requirements:**
- Items must be comparable within a single evaluative dimension (e.g., "message appeal" or "feature importance")
- Typical item count: 15–30 items
- Balanced Incomplete Block (BIB) design: each item appears in equal number of tasks
- Standard format: 4 items per set, 12–15 tasks per respondent
- Config requires:
  - `item_codes`: `{1: 'A1', 2: 'A2', ...}` (1-indexed)
  - `item_text`: `{1: 'Full text of item...', ...}`
  - `canonical_sets`: the BIB design matrix (list of item-number lists)

**Column naming in data:**
- Simple schema: `Best_1`, `Worst_1`, `Best_2`, `Worst_2`, ...
- Region-prefix schema: `QMD_{region}_coded_{t}best`, `QMD_{region}_coded_{t}worst`
- Each column stores the position (1-indexed within the set) of the chosen item

**Questionnaire rule:** MaxDiff items must be written as parallel, comparable statements (same grammatical structure, similar length). Items that are not comparable on the same dimension will produce uninterpretable utilities. Design the BIB matrix before writing the questionnaire.

---

### 5. MaxDiff Head-to-Head

**Module:** `maxdiff.py` via `compute_head_to_head()`

**Pair definition requirements:**
- Each pair consists of two items from the MaxDiff design that test variant framings of the same concept
- Config requires:
  ```python
  config['maxdiff']['paired_variants'] = [
      {'code': 'A2', 'theme': 'Daily Life Impact',
       'a': 2, 'a_label': 'Emotional framing',
       'b': 3, 'b_label': 'Rational framing'},
  ]
  ```
- Item numbers are 1-indexed and must match `item_codes`

**Questionnaire rule:** If head-to-head comparison is planned, include a/b variant pairs in the MaxDiff item list (e.g., item 2 = emotional framing of concept A, item 3 = rational framing of concept A). Both variants must appear in the same BIB design.

---

### 6. MaxDiff Bundle Evaluation

**Module:** `maxdiff.py` via `compute_bundle_metrics()`, `compute_bundle_degradation()`

**Bundle definition requirements:**
- Each bundle is a pre-defined group of 3–5 MaxDiff items
- Items are identified by their 1-indexed item numbers
- Config requires:
  ```python
  config['maxdiff']['bundles'] = [
      {'name': 'Bundle 1', 'items': [5, 12, 19, 23],
       'labels': ['A4', 'B5', 'C8', 'D2']},
  ]
  ```
- Bundle items must exist in the MaxDiff design

**Questionnaire rule:** No additional questionnaire requirements. Bundles are analytical constructs defined post-design. However, the MaxDiff item set should include items that can logically be grouped into bundles (e.g., items from different benefit categories that could form a "value proposition").

---

### 7. MaxDiff Regression

**Module:** `maxdiff.py` via MaxDiff regression functions

**Config requirements:**
```python
config['maxdiff']['regression'] = {
    'dv_col': 'outcome_var',
    'dv_label': 'Behavioral outcome',
    'motivation_iv_cols': ['motivation_r1', 'motivation_r2', ...],
    'motivation_label': 'Message Motivation Rating (1-5)',
    'use_maxdiff_ivs': True,
}
```

**Questionnaire rule:** If MaxDiff regression is planned, the survey must include:
1. The MaxDiff exercise itself (items + BIB design)
2. A behavioral outcome DV (e.g., purchase intent, likelihood to recommend) — single numeric variable
3. Optionally, a motivation rating battery where respondents rate the same items on a separate Likert scale (stated preference vs. revealed preference comparison)

---

### 8. Ridge Regression (Multi-DV Driver Analysis)

**Module:** `regression.py` via `run_multi_dv_driver_analysis()`

**IV/DV requirements:**
- IVs: Attitudinal battery (same items used for clustering work well)
- DVs: Behavioral outcome variables (purchase, usage, loyalty, etc.)
- Both must be numeric
- Config requires:
  - `iv_vars`: list of column names
  - `iv_labels`: dict of human-readable labels
  - `dv_vars`: list of column names
  - `dv_labels`: dict of human-readable labels
  - `alpha`: Ridge parameter (default 0.1)

**Questionnaire rule:** Include at least 3–5 behavioral outcome questions (purchase intent, usage frequency, brand loyalty, recommendation likelihood, satisfaction) coded on numeric scales. These serve as DVs with the attitudinal battery as IVs.

---

### 9. Segment Banner Variable

**Used by:** All cross-tab and regression functions

**Requirements:**
- One column with clean integer codes (1, 2, 3, 4 — not strings)
- Each code maps to a segment label via `segment_labels = {1: 'Segment A', ...}`
- Must be assigned to every respondent (no missing values in the banner variable)

**Questionnaire rule:** The segment variable is typically derived post-fielding via K-Means clustering. However, the questionnaire must include the attitudinal battery that produces it. For pre-existing segments (e.g., customer tier), include the classification question with integer-coded responses.

## Questionnaire Structure Checklist

Before finalizing any generated questionnaire, verify:

| # | Check | Constraint |
|---|---|---|
| 1 | All closed-ended questions have integer response codes | Cross-tabs, regression |
| 2 | Rating batteries use a single consistent scale | K-Means, VarClus, T2B, means |
| 3 | Attitudinal battery has 15+ items on uniform Likert scale | K-Means, VarClus |
| 4 | Multi-select items are designed for binary coding | Cross-tabs (multi) |
| 5 | MaxDiff items are comparable on one dimension | MaxDiff HB |
| 6 | BIB design parameters are defined | MaxDiff HB |
| 7 | Behavioral outcome DVs are included (3–5 questions) | Ridge regression |
| 8 | T2B codes are defined for every rating battery | Cross-tabs (T2B) |
| 9 | Variable naming follows prefix convention | Cross-tabs, config building |
| 10 | No mixed scales within a single battery | All numeric analyses |

## Variable Naming Conventions

Consistent naming makes config-building straightforward:

| Question Type | Convention | Example |
|---|---|---|
| Single-select | `Q{section}{number}` | `QD1` (Demographics Q1) |
| Multi-select items | `Q{section}{number}r{item}` | `QB2r1`, `QB2r2` |
| Rating battery items | `Q{section}{number}r{item}` | `QA1r1`, `QA1r2` |
| MaxDiff best/worst | `Best_{task}` / `Worst_{task}` | `Best_1`, `Worst_1` |
| Derived variables | `D_{name}` | `D_segment`, `D_age_group` |

## Scale Definition Reference

Standard scales that work with the analytic pipeline:

**5-Point Agreement (recommended for attitudinal batteries):**
```
1 = Strongly Disagree
2 = Somewhat Disagree
3 = Neither Agree nor Disagree
4 = Somewhat Agree
5 = Strongly Agree
t2b_codes = [4, 5]
```

**7-Point Agreement (alternative for finer granularity):**
```
1 = Strongly Disagree
2 = Disagree
3 = Somewhat Disagree
4 = Neither Agree nor Disagree
5 = Somewhat Agree
6 = Agree
7 = Strongly Agree
t2b_codes = [6, 7]
```

**5-Point Frequency:**
```
1 = Never
2 = Rarely
3 = Sometimes
4 = Often
5 = Always
t2b_codes = [4, 5]
```

**5-Point Likelihood:**
```
1 = Not at all likely
2 = Not very likely
3 = Somewhat likely
4 = Very likely
5 = Extremely likely
t2b_codes = [4, 5]
```

**5-Point Satisfaction:**
```
1 = Very dissatisfied
2 = Somewhat dissatisfied
3 = Neither satisfied nor dissatisfied
4 = Somewhat satisfied
5 = Very satisfied
t2b_codes = [4, 5]
```
